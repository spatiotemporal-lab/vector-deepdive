tictoc::tic(msg = "DuckDB | Full query: intersection + aggregation")

result_duckdb <- DBI::dbGetQuery(con, "
  WITH
    canton_areas AS (
      SELECT
        name                  AS canton_name,
        ST_Area(geom)         AS canton_area_m2,
        geom                  AS canton_geom
      FROM kantone
    ),
    forest_per_canton AS (
      SELECT
        c.canton_name,
        c.canton_area_m2,
        SUM(ST_Area(ST_Intersection(w.geom, c.canton_geom))) AS forest_area_m2
      FROM wald         AS w
      JOIN canton_areas AS c
        ON ST_Intersects(w.geom, c.canton_geom)
      GROUP BY c.canton_name, c.canton_area_m2
    )
  SELECT
    canton_name,
    ROUND(canton_area_m2  / 1e6, 2) AS canton_area_km2,
    ROUND(forest_area_m2  / 1e6, 2) AS forest_area_km2,
    ROUND(forest_area_m2 / canton_area_m2 * 100, 2) AS forest_pct
  FROM forest_per_canton
  ORDER BY forest_pct DESC
")

Discussion

I had no prior experience with either sf or DuckDB before this course, so getting the code to run at all was already a challenge. One thing that surprised me early on was just how long loading the data takes — filtering the forest polygons alone took over 80 seconds, which I did not expect for what felt like a simple step.

Looking at the results, Ticino stands out with over 42% forest cover. This makes sense geographically since it is a mountainous, heavily forested canton. On the other end, Geneva and Basel-Stadt are below 12%, which also makes sense as they are mostly urban. I had not expected the difference between cantons to be quite so large.

For the timing comparison, DuckDB was only about 1.2–1.3x faster than sf, which is less than I would have expected. After looking into it, I learned that both tools actually use the same underlying GEOS library for the geometry calculations — so the bottleneck is the same. The difference would likely be larger with a much bigger dataset, since DuckDB can stream data without loading everything into memory at once, while sf has to hold it all in RAM.

Task 3 – DE-9IM: King’s case & Bishop’s case

The DE-9IM (Dimensionally Extended 9-Intersection Model) is a way to describe the spatial relationship between two geometries. It looks at how the interior, boundary, and exterior of two shapes intersect, and encodes this as a 9-character string. For this task, I use it to distinguish between cells that share a full edge (like a King’s move) versus cells that only share a corner (like a Bishop’s move).

I(B) B(B) E(B)
I(A) [0] [1] [2]
B(A) [3] [4] [5]
E(A) [6] [7] [8]

Symbols: F = empty, 0 = point, 1 = line, 2 = area, * = any.

The key position is [4] — the boundary-boundary intersection — which tells us whether two cells share an edge (line = 1) or only a corner (point = 0).

rel_matrix <- sf::st_relate(centre_sf, dest_sf, sparse = FALSE)

results <- lapply(seq_len(nrow(dest_sf)), function(i) {
  deim_str <- rel_matrix[1, i]
  bb_dim   <- substr(deim_str, 5, 5)
  rel <- dplyr::case_when(
    bb_dim == "1" ~ "edge-adjacent",
    bb_dim == "0" ~ "corner-adjacent",
    TRUE          ~ "other"
  )
  data.frame(cell = i, `DE-9IM` = deim_str,
             `BB dim` = bb_dim, type = rel,
             check.names = FALSE)
})

Manual derivation

(a) King’s case – all adjacent cells (edge + diagonal)

Edge-adjacent cells share a full side → B(A)∩B(B) is a line (1):

        I(B)  B(B)  E(B)
I(A)  [  F  |  F  |  2  ]
B(A)  [  F  | *1* |  1  ]
E(A)  [  2  |  1  |  2  ]

→ DE-9IM = FF2F11212

  • I∩I = F: the interiors of two touching polygons don’t overlap.
  • B(A)∩B(B) = 1: the shared edge is part of both boundaries → line.
  • I(A)∩E(B) = 2: A’s interior is entirely outside B → area.

Corner-adjacent cells only share a single point → B(A)∩B(B) is a point (0):

        I(B)  B(B)  E(B)
I(A)  [  F  |  F  |  2  ]
B(A)  [  F  | *0* |  1  ]
E(A)  [  2  |  1  |  2  ]

→ DE-9IM = FF2F01212

King’s case pattern — covers both edge and corner neighbours using a wildcard:

→ FF2F*1212

(b) Bishop’s case – diagonal cells only

The Bishop only moves diagonally, so only corner-adjacent cells count. Position [4] must be 0 (point) to exclude edge-adjacent cells:

→ FF2F01212

Reflection

Task 3 was conceptually the most challenging part for me. The DE-9IM string is not something I had encountered before, and it took a while to understand what the 9 positions actually represent.

What helped was focusing on position [4] — the boundary-boundary intersection. Once I understood that this single character is what distinguishes a shared edge (line = 1) from a shared corner (point = 0), the rest followed quite naturally.

What I found interesting is that the King and Bishop patterns share almost the same DE-9IM string — the only difference is that one character. This shows how precise the model is: it can distinguish two very similar spatial relationships just from that one position.

If I were to use this in practice, I could see it being useful for things like network analysis — for example, checking whether two road segments truly share an edge or only touch at a point, which would affect routing differently.
