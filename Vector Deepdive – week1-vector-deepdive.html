WITH
  canton_areas AS (
    SELECT
      {name_kant}                    AS canton_name,
      ST_Area({geom_kant})           AS canton_area_m2,
      {geom_kant}                    AS canton_geom
    FROM kantone
  ),
  forest_per_canton AS (
    SELECT
      c.canton_name,
      c.canton_area_m2,
      SUM(ST_Area(ST_Intersection(w.{geom_wald}, c.canton_geom))) AS forest_area_m2
    FROM wald          AS w
    JOIN canton_areas  AS c
      ON ST_Intersects(w.{geom_wald}, c.canton_geom)

SELECT
  canton_name,
  ROUND(canton_area_m2  / 1e6, 2) AS canton_area_km2,
  ROUND(forest_area_m2  / 1e6, 2) AS forest_area_km2,
  ROUND(forest_area_m2 / canton_area_m2 * 100, 2) AS forest_pct
FROM forest_per_canton
ORDER BY forest_pct DESC
"""

Discussion

In this run DuckDB was only marginally faster than GeoPandas (~1.6×), which is less than one might expect. This is worth discussing:

Why the speedup is modest here:

1 – Shared geometry kernel. The bottleneck is the geometry computation itself (ST_Intersection, ST_Area), which relies on the same underlying GEOS C++ library in both tools. Neither has a fundamental advantage at the polygon-clipping level.

2 – GeoPandas is well-optimised. gpd.overlay() uses a spatial index (STRtree) internally, so it is not naive row-by-row iteration.

3 – DuckDB spatial is relatively new. For pure polygon-clipping workloads it does not yet dramatically out-perform GEOS-backed GeoPandas.

Why DuckDB wins at larger scale:

  • For datasets orders of magnitude larger, DuckDB streams data lazily from disk without loading everything into RAM — GeoPandas requires the full dataset in memory upfront.
  • DuckDB’s multi-threaded columnar engine scales better for aggregations over millions of features.
  • A single SQL query avoids multiple intermediate GeoDataFrames that GeoPandas must materialise in memory.

Conclusion: For this dataset size (~137k polygons, 26 cantons) both tools are roughly equivalent. DuckDB’s advantage becomes clear at scale or when combining spatial queries with complex SQL aggregations.

Task 3 – DE-9IM: King’s case & Bishop’s case

The Dimensionally Extended 9-Intersection Model (DE-9IM) encodes the topological relationship between two geometries A and B as a 3×3 matrix — intersections of their interior (I), boundary (B) and exterior (E) — read row-major as a 9-character string:

I(B) B(B) E(B)
I(A) [0] [1] [2]
B(A) [3] [4] [5]
E(A) [6] [7] [8]

Symbols: F = empty, 0 = point, 1 = line, 2 = area, T = non-empty, * = any.

Position [4] — B(A)∩B(B) — is the key discriminator for chess adjacency.

Manual derivation

(a) King’s case – all adjacent cells (edge + diagonal)

Edge-adjacent cells share a full side → B(A)∩B(B) is a line (1):

        I(B)  B(B)  E(B)
I(A)  [  F  |  F  |  2  ]
B(A)  [  F  | *1* |  1  ]
E(A)  [  2  |  1  |  2  ]

→ DE-9IM = FF2F11212

  • I∩I = F: open interiors of two touching polygons are disjoint — they don’t overlap.
  • I(A)∩B(B) = F: A’s open interior does not cross B’s boundary (the shared edge belongs to both boundaries, not A’s interior).
  • B(A)∩I(B) = F: symmetrically, B’s boundary does not enter A’s interior.
  • B(A)∩B(B) = 1: the shared edge is part of both boundaries → line intersection.
  • I(A)∩E(B) = 2: A’s interior lies entirely in B’s exterior → area.

Corner-adjacent cells share only one corner → B(A)∩B(B) is a point (0):

        I(B)  B(B)  E(B)
I(A)  [  F  |  F  |  2  ]
B(A)  [  F  | *0* |  1  ]
E(A)  [  2  |  1  |  2  ]

→ DE-9IM = FF2F01212

  • B(A)∩B(B) = 0: only the single shared corner → point intersection.
  • I(A)∩B(B) = F: the corner is on A’s boundary, not its open interior.

King’s case pattern — matches both sub-cases using a wildcard at position 4:

→ FF2F*1212

  • Positions 0–3 = FF2F: identical in both cases.
  • Position 4 (B∩B) = *: accepts both 1 (edge) and 0 (corner).
  • Positions 5–8 = 1212: identical in both cases.

(b) Bishop’s case – diagonal cells only

The Bishop moves only diagonally → only corner-adjacent cells qualify.
The DE-9IM must exclude edge-adjacent (BB = 1) and include only corner-adjacent (BB = 0):

→ FF2F01212

The discriminator is position [4] = 0 (point), which is impossible for edge-adjacent cells.
